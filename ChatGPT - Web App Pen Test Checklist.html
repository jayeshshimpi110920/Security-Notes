A checklist in web application penetration testing is a structured guide of tasks and steps to ensure comprehensive testing. It helps testers systematically cover all possible attack vectors and security flaws, making the process more organized and thorough. Here’s a basic checklist for web application penetration testing:

1. Information Gathering

  • Identify technologies used (server, language, database, frameworks).
  • Map the application (subdomains, endpoints, hidden files).
  • Review input points (parameters, form fields, APIs).

2. Authentication Testing

  • Test login forms for brute-force attacks.
  • Check for weak authentication mechanisms.
  • Test for multi-factor authentication implementation.

3. Session Management

  • Verify session tokens for randomness and security (e.g., not predictable).
  • Test for session fixation or hijacking vulnerabilities.
  • Ensure proper session expiration on logout or inactivity.

4. Input Validation

  • Test for SQL injection vulnerabilities.
  • Test for cross-site scripting (XSS).
  • Check for server-side request forgery (SSRF).

5. Business Logic Testing

  • Test for any flaws in business logic that could allow for privilege escalation or bypassing workflows.
  • Test for proper handling of edge cases.

6. File Uploads

  • Test file uploads for malicious file types or extensions.
  • Ensure proper file validation and content checking.

7. Error Handling

  • Check if error messages disclose sensitive information.
  • Test for stack traces and debug information in error responses.

8. Access Control

  • Test for insecure direct object references (IDOR).
  • Check for proper role-based access control (RBAC) implementation.
  • Test for horizontal and vertical privilege escalation.

9. Security Headers

  • Check for missing security headers (CSP, X-Content-Type-Options, HSTS).
  • Verify that headers are properly configured.

10. API Testing

  • Test APIs for broken authentication and excessive data exposure.
  • Check API responses for sensitive data leaks.
  • Test for improper rate limiting or DoS vulnerabilities.

11. Cryptography

  • Ensure strong encryption algorithms are used for sensitive data (both in transit and at rest).
  • Test for improper storage of credentials or sensitive information.

12. Miscellaneous

  • Check for common vulnerabilities (CSRF, Open Redirect, etc.).
  • Test for security misconfigurations (e.g., directory listing enabled).

This checklist serves as a guideline for testing different areas of a web application, helping you ensure that no critical area is left untested. You can adjust it based on the specifics of the web application you're testing.

When discussing security headers, there are several key headers that web applications should implement to enhance security and prevent vulnerabilities like clickjacking, XSS (Cross-Site Scripting), and information leakage. Below is a comprehensive list of essential security headers, the associated values, and how they contribute to securing web applications:

1. Strict-Transport-Security (HSTS)

  • Purpose: Enforces the use of HTTPS by telling browsers to only interact with the site over secure connections.
  • Recommended Value:
    plaintext
    Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
  • Breakdown:
    • max-age=31536000 (1 year): The time, in seconds, that browsers should remember to only access the site over HTTPS.
    • includeSubDomains: Applies this rule to all subdomains.
    • preload: Instructs browsers to preload the domain in their HSTS lists.

2. X-Frame-Options

  • Purpose: Prevents the site from being embedded in an iframe, protecting against clickjacking attacks.
  • Recommended Value:
    plaintext
    X-Frame-Options: DENY
  • Options:
    • DENY: Completely disallows the site to be framed.
    • SAMEORIGIN: Allows framing only by the same origin.

3. Content-Security-Policy (CSP)

  • Purpose: Mitigates XSS (Cross-Site Scripting) and other code injection attacks by specifying allowed sources for content like scripts, styles, and media.
  • Recommended Value:
    plaintext
    Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' https://trusted.cdn.com; object-src 'none'
  • Breakdown:
    • default-src 'self': Allows all resources to be loaded only from the same origin.
    • script-src 'self' 'unsafe-inline' https://trusted.cdn.com: Limits JavaScript sources to the same origin and trusted CDNs (but 'unsafe-inline' should ideally be avoided).
    • object-src 'none': Disallows the use of <object>, <embed>, and <applet> tags.

4. X-XSS-Protection

  • Purpose: Enables the browser’s built-in XSS protection filter.
  • Recommended Value:
    plaintext
    X-XSS-Protection: 1; mode=block
  • Breakdown:
    • 1: Enables the XSS filter.
    • mode=block: Tells the browser to block the response if an XSS attack is detected.

5. X-Content-Type-Options

  • Purpose: Prevents the browser from MIME-type sniffing, reducing the risk of drive-by downloads and script injection.
  • Recommended Value:
    plaintext
    X-Content-Type-Options: nosniff
  • Breakdown:
    • nosniff: Forces browsers to respect the MIME types set by the server, preventing them from interpreting files as different types (e.g., treating a .txt file as .html).

6. Referrer-Policy

  • Purpose: Controls the amount of referrer information sent along with requests to external domains.
  • Recommended Value:
    plaintext
    Referrer-Policy: no-referrer
  • Breakdown:
    • no-referrer: No referrer information is sent.
    • Other options include strict-origin-when-cross-origin or same-origin, depending on the level of referrer data sharing desired.

7. Permissions-Policy (formerly Feature-Policy)

  • Purpose: Controls which browser features (like geolocation, camera, microphone) can be used by the site.
  • Recommended Value:
    plaintext
    Permissions-Policy: geolocation=(), microphone=(), camera=(), fullscreen=*
  • Breakdown:
    • geolocation=(): Disables geolocation for the site.
    • microphone=(): Disables microphone access.
    • camera=(): Disables camera access.
    • fullscreen=*: Allows fullscreen access for all origins (can be restricted).

8. Expect-CT

  • Purpose: Enforces Certificate Transparency, helping prevent the use of fraudulent certificates.
  • Recommended Value:
    plaintext
    Expect-CT: max-age=86400, enforce, report-uri="https://yourdomain/report"
  • Breakdown:
    • max-age=86400: Sets the time period (in seconds) for browsers to enforce Certificate Transparency.
    • enforce: Tells browsers to reject connections that violate Certificate Transparency.
    • report-uri: Specifies a URL where browsers can send reports of violations.

9. Cache-Control

  • Purpose: Prevents sensitive data from being cached by the browser or intermediaries.
  • Recommended Value:
    plaintext
    Cache-Control: no-store, no-cache, must-revalidate
  • Breakdown:
    • no-store: Prevents any caching of the resource.
    • no-cache: Forces revalidation before serving a cached resource.
    • must-revalidate: Ensures that once a resource becomes stale, it must be validated by the origin server before being used again.

11. Cross-Origin-Embedder-Policy (COEP)

  • Purpose: Ensures that a document can only load resources (e.g., scripts) from the same origin or explicitly allowed origins.
  • Recommended Value:
    plaintext
    Cross-Origin-Embedder-Policy: require-corp
  • Breakdown:
    • require-corp: Requires resources to have the CORP header to allow embedding.

12. Cross-Origin-Opener-Policy (COOP)

1. Checking for Security Misconfiguration (OWASP A05:2021)

Understanding Security Misconfiguration

Security misconfiguration occurs when security settings are not defined, implemented, or maintained properly. This can include default configurations, incomplete configurations, open cloud storage, misconfigured HTTP headers, error messages containing sensitive information, and more.

How to Check for Misconfigurations

Using automated tools can help quickly identify common misconfigurations.

    • Usage: Run a scan against your target systems to identify vulnerabilities and misconfigurations.
  • OpenVAS
    • Purpose: An open-source vulnerability scanner that identifies misconfigurations and security issues.
    • Usage: Set up OpenVAS and perform a full system scan to detect misconfigurations.
  • Nikto
    • Purpose: A web server scanner that tests for dangerous files/CGIs, outdated server software, and misconfigured server options.
    • Usage: Run Nikto against the target web server to find potential misconfigurations.

b. Manual Configuration Review

Manually reviewing system configurations can uncover issues that automated tools might miss.

  • Review Server Configuration Files
    • Apache: httpd.conf, .htaccess
    • Nginx: nginx.conf
    • IIS: Web.config files

      • Default accounts and passwords.

  • Check Application Configurations
    • Ensure that debug modes are disabled.
    • Verify that proper error handling is implemented without exposing stack traces.

  • Insecure Permissions: Files or directories with permissions that allow unauthorized access.
